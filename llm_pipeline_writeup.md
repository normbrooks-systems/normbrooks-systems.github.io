# A Wiki Generator That Actually Cites Its Sources

## What this is

I spent nine days building a pipeline that takes a novel and produces a queryable database of its lore — characters, vessels, locations, factions, technology, events — plus generated wiki articles for each entity, with provenance traceable back to specific chapters and scenes in the source text. The calibration corpus was David Weber's *On Basilisk Station*, the first Honor Harrington novel. The output isn't perfect, but the architecture works and the imperfections are honest about where they live.

This writeup has three parts. First, this section — a quick preview to give you an idea of what's coming. Next, a look at what the system is doing during the process. And finally, a postmortem on what I learned building it — which is the part I think is actually worth your time.

## The shape of the system

The pipeline runs in phases. A summarization layer at the front converts each scene of the source text into a short paragraph capturing who's there, where it is, and what happens. Extraction passes run against those summaries, identifying named entities and writing them into per-type tables in a SQLite database.

A consolidation step deduplicates the extracted names and resolves aliases. Each character record stores first, middle, last, and maiden names as separate fields, and titles like "Captain" or "Dame" are stripped at extraction time against a known title list. So when the prose says "Captain Harrington," the system strips the rank and matches "Harrington" against the last name on the character record. The aliases table holds genuine alternate identities — nicknames, callsigns, and name variants that don't decompose into the formal name structure — rather than every honorific-prefixed mention.

Once the entity tables are populated, fill passes go back to the source text in narrow snippets to populate additional fields — species, gender, ship class, location type, parent organization, and so on. A description pass extracts physical and characterological details across the whole book, distinguishing permanent traits from momentary ones. A scene-mapping pass walks the database and builds junction tables linking each scene to every entity that appears in it. A character-status pass tracks rank and title changes across the narrative, recording when a rank was held at the start of the book and when (if ever) it changed.

The database is the substrate. Every entity has provenance back to the chapter and scene where it was first seen, every alias is tracked, every rank change is timestamped to its narrative event. From there, a generation pass assembles the data for any given entity into a structured wiki article — overview, physical description, rank history, narrative appearances broken by arc, associated vessels, key locations. The articles aren't publication-ready, but they're closer than I expected — close enough that a competent editor could clean one up in roughly the time it takes to read it carefully.

The whole thing runs locally on consumer hardware. No cloud APIs, no source text stored in the database after extraction completes, no copyrighted output that could leak into a model's training data. The legal posture was deliberate from the start: the system reads the book, learns its structure, and produces structured output that paraphrases and synthesizes rather than reproduces.

The output was validated against the existing Honorverse wiki, which has been maintained by fans for decades. The system found entities the wiki had, missed a few, invented none that mattered, and surfaced rank-change details that the wiki had documented through manual research. That's the bar I was building against.

## How the pipeline got built

The original pipeline had four or five phases, depending on how you count. Extraction passes per entity type, a consolidation step, fill passes for additional fields, then wiki article generation. Each extraction pass would feed the model a chapter's text and ask it to identify named characters, or vehicles, or locations, or whatever the pass was looking for. Scene summaries were on the roadmap — a layer that would summarize each scene before extraction ran — but they were sitting deep in the sequence, somewhere around step ten or twelve. The plan was to get the rest of the pipeline working first and add summaries as a refinement later.

That's not how it went.

The first pass against raw chapter text produced output that was technically structured but mostly wrong. The extraction model hallucinated entities that didn't exist, missed entities that did, and confidently misclassified ones it found. Three failures stand out as representative. The model kept identifying humans as Medusans — confusing the alien species native to the planet Medusa with the human characters operating there. It extracted HMS Fearless as a character rather than a vessel, presumably because the ship is referred to by name as an actor in scenes. And when I tried character physical description extraction, Honor Harrington came back with six limbs and a tail. That last one wasn't from the prose; the model invented it.

Each failure pointed at the same problem. The prose carried too much noise for the model to filter accurately. A chapter contains scenery, transitional description, internal monologue, military jargon, dialogue tags, and incidental references — alongside the entities and events that actually matter. Asking the model to extract clean named entities from that substrate was asking it to do two jobs at once: identify what's important and identify what type it is. It was failing at both.

I started moving the summary layer up. Not all at once — incrementally, one extraction pass at a time. Whichever pass was producing the worst output got modified to run against a generated short summary of each scene instead of against raw chapter text. The change worked. Each extraction pass that got migrated produced cleaner output immediately. So the next worst-performing pass got migrated. Then the next.

By the time this stabilized, every extraction pass was running against summaries, the summary layer was at position 1 in the pipeline, and everything downstream had been restructured to consume summaries as its input substrate. What had been step ten or twelve was now step one. The roadmap had inverted under pressure.

The qualitative win was unmistakable. Before the migration, my entity extraction passes produced output where I was deleting a substantial fraction of entries during human review — junk entities, hallucinations, miscategorizations, duplicates that should never have been emitted in the first place. After the summary layer was at position 1, the noise dropped by roughly an order of magnitude. An order of magnitude reduction in human cleanup work, just from putting a summarizer in front of everything else. The summaries were doing two filters: they pruned the noise from raw prose, and they collapsed multiple references to the same entity into single mentions, so the extraction model wasn't being asked to handle deduplication implicitly.

That migration wasn't the only one. The pattern repeated. Tasks that had originally been single phases kept fissioning into multiple smaller phases as I built out the pipeline. Phases I had originally scoped as one step kept turning into several. Every time I tried to do too much in one model call, the output degraded. The fix was always to break the task smaller. By the time the pipeline stabilized, several of the original phases had each fissioned into roughly a half-dozen specialized scripts, plus review tooling, plus secondary passes at higher model tiers for whatever the primary passes left incomplete.

There were two distinct pressures pushing the architecture in the same direction. The model couldn't hold complex tasks without hallucinating, so I kept fissioning tasks. And Claude — which I was using as my AI development partner throughout this build — couldn't hold the project's complexity in a single context window either. I hit Opus's context limit twice in one day during the worst of it, and averaged well over a million tokens daily across parallel chats focused on different subsystems. Both forces wanted smaller, more focused units of work. I gave them what they wanted because the alternative was watching either system collapse under the load.

The pipeline architecture you'd see if you read it now isn't a designed shape. It's the residue. Each phase boundary is a place where one task fissioned into smaller tasks because the combined version was producing garbage. The architecture isn't optimized for elegance. It's optimized for the failure modes I kept hitting. Every place where two phases could conceivably be one phase is a place I tried that and watched it produce worse output.

The pattern, abstracted: when an LLM task produces consistently bad output, the first move isn't to make the model bigger or the prompt cleverer. The first move is to ask whether the task is actually one task or several tasks bundled together. Most of the time, in this project, it was several. Splitting them gave each component a smaller, more focused job that the model could handle. The cost was more pipeline machinery — more scripts, more intermediate database tables, more orchestration. The benefit was output that didn't need to be thrown away.
